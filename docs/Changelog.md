# Changelog

All notable changes to this project will be documented in this file. This project tries to use [semantic versioning](https://semver.org/).

## 2.0.0 (2026-07-12)
  * Breaking change: `EspHomeClient` no longer extends `EventEmitter`. Subscribe via the new composition API instead: `client.on(event, handler)` returns a `Disposable`, `client.once(event, options?)` returns a `Promise`, and `client.stream(event, options?)` returns an `AsyncIterable` with backpressure policy.
  * Breaking change: the `send*Command` wrapper methods are gone. Use `client.command(id, options)` with a branded `EntityId<T>` minted via `entityId(type, objectId)`. The branded type carries the entity type tag at compile time so passing a switch id to a light command is a type error.
  * Breaking change: top-level voice-assistant, Home Assistant, user-services, and camera methods are gone in favor of lazy sub-APIs reached through `client.voiceAssistant`, `client.homeAssistant`, `client.services`, and `client.camera(id)`. Method names map directly (`subscribeHomeAssistantServices` -> `client.homeAssistant.subscribeServices`, `sendCameraImageRequest` -> `client.camera(id).snapshot()`, and so on).
  * Breaking change: numeric bitmask fields are replaced by named booleans on telemetry and command surfaces. `ClimateEntity.featureFlags: number` becomes six named booleans (`supportsCurrentTemperature`, `supportsAction`, etc.); `WaterHeaterEvent.state: number` and the water-heater command's `state: number` become independent `awayState` and `onState` booleans.
  * Breaking change: the v1 `export enum` declarations are now `export const X = { ... } as const` plus a derived literal-union type. Forward access (`X.MEMBER === value`) is unchanged. TypeScript's reverse-mapping idiom (`X[someNumber]` for name lookup) is gone; use the relevant `*Name(value)` helper or build a reverse map at the call site.
  * Breaking change: the v1 connection state-machine enums (`Handshake`, `ConnectionState`, `ProtocolType`) are gone. Observe connection transitions via `client.lifecycle()` or `client.healthStream()` instead.
  * Breaking change: `NoiseHandshakeError` moved from `esphome-client/crypto-noise` to `esphome-client/errors`. Root-package imports are unchanged; sub-path imports must update.
  * Breaking change: minimum supported Node.js is 22.20; minimum regression-tested ESPHome firmware is 2025.10.
  * Breaking change: `client.health()` returns a `ConnectionHealth` discriminated union. `LiveConnectionHealth` (state `connected` or `stalled`) carries the connect epoch `connectedAtMs`; `DownConnectionHealth` (state `disconnected` or `reconnecting`) omits it and is always `encrypted: false`. The flat `uptimeMs` field is removed - derive live uptime with the new `connectionUptimeMs(health)` and narrow with `isConnectionLive(health)`. Uptime now stays correct through a stall, which the previous flat record reported as `0`.
  * New feature: three new entity types - `water_heater`, `infrared`, and `radio_frequency` - lifting the registry from 23 to 26. The IR and RF types support raw-timings transmit via `client.transmitRawTimings(id, options)` and inbound receive events; capability detection flows through the `InfraredCapabilityFlags` and `RadioFrequencyCapabilityFlags` bitmasks.
  * New feature: branded `EntityId<T>` types and schema-derived public shapes. `Entity`, `TelemetryEvent`, `EntityFor<T>`, `StateEventFor<T>`, and `CommandFor<T>` are derived from `ENTITY_SCHEMAS`; wire-enum fields on both state and discovery now narrow to literal unions tied to the matching named constants (`LockState`, `CoverOperation`, `EntityCategory`, etc.). Exhaustive switches over state events compile-check.
  * New feature: typed error hierarchy with `cause` chains covering encryption, handshake, connection, protocol, operational, and configuration paths. The `PermanentError` marker class is what auto-reconnect filters on. Resource bounds (`maxFrameBytes`, `maxFieldsPerMessage`, `maxRecvBufferBytes`) protect the decoder from malformed or hostile input and surface typed errors on breach.
  * New feature: auto-reconnect with exponential backoff (500ms initial, 2x, 30s cap, 20% jitter). `PermanentError` subclasses short-circuit the loop. Consumer subscriptions, the latest-state cache, and event-bus listeners survive each reconnect; iterators pause during disconnect and resume on the next connect. Pass `reconnect: false` to disable.
  * New feature: lazy heartbeat (default 30s idle / 60s stall). Stalls trigger `HeartbeatStalledError` and a graceful disconnect. Pass `keepAlive: false` to opt out.
  * New feature: typed health and lifecycle observability. `client.health()` returns a synchronous `ConnectionHealth` snapshot; `client.healthStream({ signal })` and `client.lifecycle({ signal })` provide async-iterable observation over transitions and tagged `LifecycleEvent` records.
  * New feature: optional observability metrics. Wire a `metrics` implementation of the `ClientMetrics` contract (`increment` / `timing` / `gauge`) to a backend like StatsD, OpenTelemetry, or Prometheus, and the library emits low-cardinality frame, connection, reconnect, heartbeat-RTT, command, and discovery metrics. The default - `metrics` omitted - short-circuits to zero overhead.
  * New feature: latest-state cache. `client.latest(id)`, `client.snapshot()`, and `client.snapshotFor(type)` return type-narrowed views of the most recent state event per branded entity id. The cache is updated before listeners are notified, so a read from inside a `telemetry` listener sees the event that fired it.
  * New feature: structured capability record. `client.capabilities()` returns a `ClientCapabilities` covering API minor version, encryption, voice-assistant features, Bluetooth proxy, serial proxy, Z-Wave proxy, modern handshake, and noise-key rotation support. Consumer feature gating consults this record rather than parsing version strings. ESPHome API 1.14 wire-level features (`clientDerivedObjectId`, `climateTemperatureUnit`, `lockOpenStates`, `voiceAssistant.stereoAudio`) gate via the same record.
  * New feature: Bluetooth, Z-Wave, and serial proxy sub-APIs. `client.bluetooth` exposes advertisement scanning and GATT operations (connect, pair, read/write characteristics and descriptors, notify) - BLE addresses round-trip as `bigint`. `client.zwave` is a transparent byte pipe to the device's Z-Wave Serial API; consumers route frames into a Z-Wave-aware library like `zwave-js`. `client.serial` bridges UART instances with per-instance configure, read, write, flush, and modem-pin control.
  * New feature: `client.commandAndAwait(id, options, awaitOptions?)` subscribes to the matching state-event channel before sending the command (avoiding the fast-device race) and resolves with the first matching event. Rejects with `DOMException("TimeoutError")` after the default 2000ms or the consumer's `AbortSignal`.
  * New feature: graceful disposal via the explicit-resource-management proposal. `await using client = await openEspHomeClient(options)` opens a client and sends `DISCONNECT_REQUEST` on scope exit, waiting briefly for the device acknowledgment; `using sub = client.on(...)` does the same for subscriptions.
  * New feature: Web Streams interop. `client.telemetryReadable()`, `client.logsReadable()`, `client.lifecycleReadable()`, `client.voiceAssistant.audioReadable()`, and `client.camera(id).readable()` return `ReadableStream`s for `pipeThrough` / `tee` / `pipeTo` composition.
  * New feature: schema-extension API. Downstream consumers can register vendor-specific entity types via the `extraSchemas` option. `aliasOf("cover")` mirrors an upstream type under a different type tag; `extending("switch", { addedListEntitiesFields, addedStateFields })` adds read-side fields to an upstream type. Built-in keys cannot be silently shadowed.
  * New feature: multi-device support. `client.subDevices()` enumerates parent-ESP sub-devices; `client.entitiesByDevice(deviceId)` filters the registry by parent device.
  * New feature: testing subpath. `esphome-client/testing` exports `MockClient` (high-level consumer-facing test harness), `MockTransport` (integration-level seam), and factory helpers (`mockEntity`, `mockStateMessage`, `mockDeviceInfo`, etc.). Every sub-API on `MockClient` is a `Proxy`-backed recording mock so tests address recorded calls and stage return values uniformly.
  * Improvement: in-flight awaits like `disconnectAsync()` and `setNoiseEncryptionKey()` reject with a typed `DOMException("AbortError")` on reset / reconnect / teardown instead of silently hanging until their internal timeouts.
  * Improvement: pre-shared keys are validated at construction. A non-32-byte (after base64 decode) PSK is logged and dropped; the client falls through to plaintext rather than failing partway through the noise handshake.
  * Improvement: warm-path inbound throughput. The per-state-event path no longer pays a `JSON.stringify` on the default no-op debug logger, emits the diagnostic `message` event only when a consumer is subscribed, memoizes the event-bus channel names, and reuses the registry's pre-minted entity id - cutting measurable per-frame work on the hot dispatch path with no behavior change.
  * Improvement: the camera sub-API bounds multi-packet image reassembly with a new `maxImageBytes` option (default 8 MiB). A device that streams an oversized image, or never sets the done flag, has the partial image dropped with a warning instead of growing the accumulator without limit; this is a per-image fault and does not drop the connection.
  * Improvement: `client.capabilities()` and the disconnected `client.health()` snapshot return boundary copies, so mutating a returned record can no longer corrupt the client's cached state (which `client.isEncrypted` reads).
  * Improvement: a run-phase message handler that throws - including a consumer `on()` listener that throws, which surfaces here through `emit` - is now logged and isolated, and the connection stays up, instead of the exception being silently swallowed. Setup-phase buffer-overflow drops are logged too.
  * Improvement (security): the Noise zeroization contract now closes on the success path. The decoded PSK, the chaining key, and the handshake hash are wiped once the cipher pair is installed, and the live ChaCha20 session keys are zeroed at teardown. The cipher also refuses the reserved maximum nonce (2^64 - 1) with a typed `NoiseHandshakeError` (`NONCE_EXHAUSTED`) per Noise Protocol Framework section 5.1. The chaining key - the HKDF input that `Split()` expands into the live session keys - is no longer written to the debug log; only a non-reversible progress signal is emitted.
  * Improvement: `espc monitor --duration` accepts fractional seconds (e.g. `1.5`), matching `watch` and `record`. `espc control` now validates device-independent argument input (mode, dates, times, `--temp`, siren `--duration`) at the boundary - before opening a connection - when the entity id encodes its type.
  * Improvement: `espc snapshot` collects state adaptively - it waits for the telemetry stream to go quiet (a brief grace for the first event, then a short quiet window, capped by a ceiling) instead of a fixed delay, so the output reflects the full post-subscribe state burst. `espc snapshot --wait <seconds>` overrides this with a deterministic fixed window.
  * Improvement: the `espc list`, `devices`, `watch`, `control`, and `info` commands return as soon as discovery completes rather than paying a fixed ~1.5-2 second wait, now that a resolved `connect()` guarantees entities, sub-devices, and device info are populated.
  * Improvement: the package root now also exports the named wire-enum constants in `api-constants` (`BluetoothScannerMode`, `MediaPlayerCommand`, `ValveOperation`, `AlarmControlPanelState`, ...), the API feature-version table and its `deviceSupports` helper (`api-feature-versions`), the `UserServicesApi` sub-API (alongside the other six already exported), and the stream backpressure options (`StreamOptions`, `StreamBackpressureMode`). Every type that appears in a public method signature is now importable from `esphome-client` by name rather than reachable only by inference through a referencing type.
  * Fix: the noise -> plaintext fallback no longer leaks a `PeerClosedDuringNoiseError` to the caller when a device responds with a plaintext indicator byte during the noise handshake. Affects devices with the encryption feature compiled in but no key configured.
  * Fix: a passive transport death mid-session (peer RST/FIN, device reboot, network drop) escalates to disconnect and auto-reconnect even with `keepAlive: false`, carrying the real typed cause instead of waiting on the heartbeat stall timer. A run-phase noise decrypt failure raises a transient `DecryptionFailedError` so the session reconnects rather than being misclassified as a permanent key error.
  * Fix: graceful disconnect and manual reconnect now cooperate with the auto-reconnect supervisor. `client.disconnectAsync()` marks the client closed and cancels any in-flight reconnect loop so it stays disconnected, and calling `client.connect()` during backoff cancels and drains that loop first so the supervisor cannot tear down the session you just established by hand.
  * Fix: an unknown enum alias on a command (a typo'd or runtime-derived `climate` / `water_heater` mode) fails closed - the encoder reports a structured failure that warn-drops for `command()` and throws for `commandAndAwait()` - instead of being coerced to wire value `0`, which is "off" for those entities and would silently command the device off.
  * Fix: `client.camera(id).snapshot()` and `.stream()` distinguish inbound images by the branded entity id, so on a multi-camera device a snapshot can no longer return a different camera's image.
  * Fix: in-flight Bluetooth GATT and serial-proxy requests reject with a typed `AbortError` at the disconnect boundary instead of lingering until their per-await timeout, and a pending `client.bluetooth.connect(address)` fails fast with a typed `ConnectionError` when the device reports a clean disconnect rather than hanging until the lifecycle timeout. Connection-scoped reads such as `client.bluetooth.isConnected(address)` report the disconnected state immediately.
  * Fix: when auto-reconnect gives up - it reaches the attempt ceiling, or your `shouldRetry` predicate returns `false` or throws - the client emits a terminal disconnect carrying the typed cause instead of leaving `client.health()` and the health stream stuck at `reconnecting`. A `shouldRetry` that throws fails closed rather than escaping as an unhandled rejection.
  * Housekeeping.

## 1.3.0 (2026-01-22)
  * New feature: extended entity metadata parsing. The `entities` event now includes all fields from ListEntities responses - things like `icon`, `deviceClass`, `unitOfMeasurement`, `stateClass`, `supportedColorModes`, `effects`, `options`, and capability flags. Each of the 22 entity types now has a dedicated TypeScript interface with type-specific properties.
  * New feature: exported entity types and enums for consumers. You can now import `Entity`, `SensorEntity`, `LightEntity`, etc., along with enums like `EntityCategory`, `StateClass`, `NumberMode`, and `TextMode`.
  * Improvement: schema-driven entity parsing. Entity field definitions are now centralized in the schema system, making future protocol changes easier to maintain.
  * Improvement: better error logging when entity discovery fails due to missing required fields.
  * Housekeeping.
